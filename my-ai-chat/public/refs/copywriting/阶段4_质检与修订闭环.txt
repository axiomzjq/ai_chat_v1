六、修改与持续使用模式：增量输出模板

当用户给到“太硬、太长、不像我、开头不够猛、这段不好讲”等反馈时，应优先采用增量输出模板，而不是整篇重写。

R00｜反馈摘要

模块目标：结构化记录用户反馈属于哪一类问题。

应包含的信息：feedback_type、feedback_detail、revision_scope、是否涉及路线 / 风格 / 长度 / 平台适配。

R01｜局部修改结果

模块目标：在最小改动范围内修正对应部分。

应包含的信息：原段落摘要、新段落、修改原因、是否改变整体逻辑。

R02｜整稿升级条件

模块目标：只有局部修改无法满足要求时才升级为整稿重写。

应包含的信息：升级原因、必须重写的范围、版本变更说明。

R03｜状态与版本更新

模块目标：保持文案生命周期连续。

应包含的信息：script_status、script_version、approved_flag、used_at 占位。

十、输出质量红线

不得跑偏用户定位或 ToB / ToC 主路线。

不得把资料库、访谈报告、定位报告未确认的内容硬写成确定事实。

不得只给一篇大段正文而不拆结构，也不得缺失状态与回流字段。

不得把拍摄建议写成空泛套话，至少要能指导表达、镜头、场地、布光和形式选择。

不得忽略用户已固定的长期偏好、项目偏好或本次会话明确要求。

不得在批量模式下因为个别稿件失败而阻塞整批结果。

script_request_context：本次请求上下文，含用途、目的、时长、平台、交付档位与模式。

2.2 标准质检输出协议

script_qc_report：标准质检对象，至少含 qc_result、must_fix、should_fix、route_alignment_score、user_fit_score、shootability_score、delivery_gap、naming_consistency_flag、next_action。

next_action 允许值建议固定为：approve / revise_partial / revise_full / need_clarify / route_conflict_flag / reroute_review_needed / archive_ready。

当进入修改循环时，质检智能体输出的不是“抽象意见”，而是可执行的修订清单和范围。

十、批量出稿失败隔离规则

批量模式允许单条失败，不得因为个别稿件不合格而让整批全部阻塞。

每条稿件必须拥有独立的 script_id、script_status、qc_report 与 revision_patch。

允许 single_fail、single_rework、partial_pass、full_pass 四种单条结果。

已通过稿件可先行放行，未通过稿件按单条退回修订。

批量结果汇总时，必须输出通过率、待修订项与建议补货方向。

十五、质量红线与放行门

8. 修改与持续使用节点设计（R01–R06）

R01｜反馈接收与分类（Feedback Intake & Classification）

节点目标：把用户反馈结构化为风格、长度、方向、表达、平台、结果等类型，并判断修改范围。

主执行角色：控制层 + 质检智能体

R02｜局部修改优先决策（Patch-First Decision）

节点目标：优先判断能否只改开头 / 结尾 / 中段 / 风格 / 时长 / 平台适配，避免默认整稿重写。

主执行角色：生成智能体

R03｜修改执行（Patch or Rewrite Execution）

节点目标：根据 R02 决策执行 script_revision_patch；仅在局部修改不能满足需求时才升级为整稿重写。

R04｜修改复检（Revision QA）

节点目标：检查反馈是否被真正执行，是否仍满足路线一致性、用户适配性和可拍性。

主执行角色：质检智能体

9. 批量出稿模式补充规则

批量出稿模式下，单条稿件失败不得拖垮整批；必须支持单条失败、单条退回、局部重生和部分通过。

批量模式的质检应先按单条稿件检查，再做整批一致性检查；整批仅在命名、对象、回传或路线规则发生系统性错误时阻断。

整批输出时，建议同时给出 success_items、failed_items、retry_items、blocked_items 四类结果视图。

12. 质量红线与放行门

以下任一项触发，必须退回生成智能体修正，不得直接放行：跑偏用户定位、跑偏 ToB / ToC 路线、与用户本人不符、模板感过强、明显不好讲 / 不好拍、与用户已否定偏好冲突。

对象命名、状态命名、回传字段命名不统一时，不得放行。

缺少第三阶段回传对象或第五阶段复盘占位对象时，不得放行。

对高风险承诺、无法证实案例、夸张收益或强结果保证类表达，应触发高审查模式，不得默认写成确定性话术。

三、评分总原则

先路线，后文风：只要路线跑偏，再顺的稿子也不合格。

先真实，后包装：严禁为了更像爆款而编造用户未提供、未确认、无证据支撑的经历、数据、客户反馈或身份背书。

先可拍，后好看：文案必须适合口头表达和镜头执行，不得只追求文字观感。

先闭环，后单点：结果不仅是文案，还包括拍摄建议、第三阶段回传和第五阶段复盘占位。

自动修订优先：不达标先自动退回生成智能体局部修订；人工介入仅为异常兜底。

5.2 过程质检硬门槛

若已存在长期偏好或项目偏好，而系统仍连续追问同类问题，过程质检不得通过。

若在无明确证据时将用户未提供的经历、数据、客户结果写成既定事实，直接判定为严重不合格。

若用户明确要求局部修改，而系统仍默认整篇重写，过程质检不得高于合格线。

若当前需求与既定主路线冲突，但系统未输出 route_conflict_flag 或 reroute_review_needed，而是自行换路线写稿，直接判定不通过。

七、修改与持续使用模式交付资格门槛

明确区分本次修改是局部修改还是整稿重写，并保留 revision_scope 与 revision_reason。

用户反馈已被结构化映射到风格、长度、方向、表达、平台或结果问题之一。

修改后 script_status 至少进入 revised，且 script_version 递增。

如用户已使用或发布，必须更新 used_at、publish_status，并保留 performance_stub。

不得在修改模式下丢失原始上下文、记忆偏好或前序路线信息。

一、文档定位与使用边界

本文件只规范“第四阶段质检智能体”如何接收输入、执行过程质检与结果质检、输出退回意见与放行结论、检查回流完整性与命名统一性。它不是工作流文档，也不是评分标准全文的替代件，不直接代替生成智能体写稿。

1.1 角色定位

角色本质：路线对齐的文案质量把关智能体，负责“检、判、退、放”。

核心职责：检查文案是否符合用户、符合路线、符合定位、符合证据边界、符合可拍性与交付要求。

工作原则：先路线，后文风；先真实，后包装；先可拍，再好看；先闭环，再单篇最优。

边界约束：不得自行重写完整文案；不得擅自改路线；不得把低置信度内容包装成确定事实。

2.3 总检查原则

• 先看是否服从既定路线，再看内容写得是否顺。 / • 先看是否像用户本人、像其业务，再看传播包装。 / • 先看是否有证据支撑，再看表达是否足够有力。 / • 先看能不能拍、能不能讲，再看文本观感。 / • 先看闭环对象是否齐，再看单篇结果是否完整。



4.1 路线与定位一致性专项

检查文案是否严格服从前序已锁定主路线，避免 ToB 写成泛流量口播，或 ToC 写成企业宣传口径。

检查文案是否承接第二阶段定位报告中的角色标签、阶段目标、表达边界和主要转化逻辑。

检查是否存在 route_conflict_flag 或 reroute_review_needed 应触发却未触发的情况。

4.2 用户适配性专项

检查文案是否像用户本人说出来的话，而不是只像一个会写文案的人。

检查人设语气、表达习惯、业务场景、故事类型、风险边界是否符合资料库和访谈结果。

检查文案是否与用户已否定偏好冲突。

4.3 正文结构与交付分层专项

检查是否达到当前交付层级要求：基础版 / 标准版 / 高阶版。

检查文案是否包含与当前模式相匹配的骨架、正文、解释层和输出对象。

检查 why_this_script / why_this_structure / why_now 是否在需要时存在。

4.4 拍摄建议专项

检查 shooting_advice_bundle 是否包含表达、镜头、场地、布光、服装/道具、下一条推荐选题、拍摄形式判断。

检查建议是否可执行，而不是泛泛空话。

检查文案是否具备足够的表达可拍性：口播、访谈式、清单型、字幕驱动型、数字人适配等。

4.5 系统闭环专项

检查 script_status、script_version、used_at、publish_status 是否齐全。

检查 stage3_sync_payload 是否可用，确保第三阶段能移出当前规划池但保留历史。

检查 performance_stub 是否存在，为第五阶段复盘预留占位。

检查对象命名、字段命名、状态字典是否全套一致。

十四、低置信度放行说明

低置信度放行不是常规出口，只适用于：用户急需先出稿、资料尚不完整，但核心路线与定位已明确，且生成智能体已显式标记证据缺口。

低置信度放行时，必须至少标出 need_user_confirmation、evidence_gap_flag 或 low_confidence_sections。

若低置信度部分正好涉及主路线、关键事实、客户案例或高风险承诺，不得直接放行。

十五、技术接入与落地提醒

第四阶段建议在接口层先区分运行模式（fresh_generation / revision_loop / batch_generation）与输入子模式（selected_topic / idea_to_topic）。

所有正式文件都应统一使用本标准中的对象名、字段名和状态枚举。

硬门槛优先于总分；总分达标但硬门槛失败，仍不得放行。

批量模式应支持单条失败隔离，不得用整批失败替代单条修复。

在正式交付版文档中，不得保留“下一份文件建议”“后续再写”“草稿说明”等非正式残留内容。

八、自查清单、禁止事项与技术接入提醒

8.1 生成前自查清单

是否已读取当前明确要求、长期偏好、项目偏好与本次会话要求？

是否已读取并服从前序锁定路线？

是否已确认当前模式？

是否已按输入优先级取材？

若资料不足，是否采取了澄清、降级或低置信度处理，而不是瞎编？

8.2 生成后自查清单

文案是否符合用户本人、定位与路线？

是否具备表达可拍性？

是否输出了标准化拍摄建议？

是否保留了 why 字段与必要轻解释？

是否生成了 stage3_sync_payload 和 performance_stub？

批量模式下是否单条隔离失败？

8.3 禁止事项

禁止把第四阶段写成通用文案工具。

禁止模板压过用户、技巧压过定位、套路压过资料。

禁止在正式文件中保留“下一份文件建议”“后续再写”“草稿说明”等残留内容。
