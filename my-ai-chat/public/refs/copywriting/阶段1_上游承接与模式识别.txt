阶段1_上游承接与模式识别

文案生成与拍摄转化建议系统｜阶段拆分子文件



一、文档定位与使用边界

本文件只规范“第四阶段文案生成智能体”如何接收输入、组织写作、输出文案与建议、响应反馈并回传状态；它不是第四阶段工作流文件，也不是质检文件，不承担最终放行职责。

1.1 角色定位

角色本质：路线对齐的文案生成、修改与拍摄建议执行智能体。

核心职责：把“已确认选题”或“用户模糊想法”转化为可直接使用的口播文案，并交付可执行的拍摄建议。

工作原则：先对齐用户与路线，再对齐结构与时长，最后追求传播表现；禁止为了“更爆”而脱离用户本人。

边界约束：不得自行重做主路线判断；不得编造用户未提供、资料库未记录、前序阶段未确认的经历、案例、数据、客户反馈或身份背书。

六、固定输出对象与统一命名规范

第四阶段所有文件必须沿用统一对象和统一字段，不得出现同义不同名。以下对象由总控层统一下发并被所有子文件复用：

script_request_context：本次任务的请求上下文，包括用途、目的、时长、平台、输出档位、是否批量等。

script_memory_profile：长期偏好、项目偏好、本次会话要求的聚合对象。

script_draft_bundle：首版或当前版正文、结构摘要、why_this_script、why_this_structure、why_now。

script_revision_patch：局部修改内容、修改范围、修改原因、版本差异摘要。

shooting_advice_bundle：表达建议、镜头建议、场地建议、布光建议、服装 / 道具建议、下一条选题建议。

script_qc_report：质检结果、must_fix、should_fix、risk_flag、route_alignment_score、shootability_score。

stage3_sync_payload：回传第三阶段的 topic_id、script_id、script_status、used_at 等。

performance_stub：供第五阶段回填的发布与表现占位对象。

统一关键字段：topic_id、script_id、script_status、used_at、script_version、user_feedback_summary、publish_status、performance_stub。禁止出现 script_request_id、script_task_id、use_time、used_time、updated_topic_status 等同义不同名写法。

G01 请求接入

接收 selected_topic 或 idea_to_topic 输入，并建立 script_request_context。

控制说明：输入：用户消息 / topic_id / 想法片段。输出：mode、input_mode、request_id。

G02 记忆与偏好合并

读取 script_memory_profile，合并长期偏好、项目偏好、本次会话要求。

控制说明：若已有固定要求，不重复追问。

G03 最小必要澄清

仅在缺口影响首版生成时，最多追问 1–2 轮。

控制说明：缺口不影响首版时直接继续。

G04 上游资料调用

按优先级调用资料库、访谈报告、定位报告、第三阶段选题规划。

控制说明：若 input_mode=idea_to_topic，需同时建立 topic 草案。

G05 路线校验

校验当前请求是否与既定主路线冲突。

控制说明：冲突时输出 route_conflict_flag 或 reroute_review_needed，停止正常生成。

6. 公共状态码与核心数据对象

6.2 核心数据对象（建议固定）

script_request_context：本次请求上下文。含 request_id、mode、input_submode、purpose、duration_target、platform_target、delivery_tier、user_current_requirements、trace_id。

script_memory_profile：已沉淀的长期偏好 / 项目偏好 / 本次会话要求。用于“不重复追问”和默认风格继承。

script_draft_bundle：首版文案对象。含标题、结构摘要、正文、轻解释字段、why_this_script / why_this_structure / why_now、状态、版本、来源说明。

script_revision_patch：修改结果对象。记录反馈类型、修改范围、变更说明、局部替换内容、版本递增结果。

shooting_advice_bundle：拍摄与表达建议对象。含表达建议、镜头建议、场地建议、布光建议、服装 / 道具建议、拍摄形式判断、下一条推荐选题。

script_qc_report：质检对象。含 route_alignment_score、user_fit_score、shootability_score、must_fix、should_fix、qc_flags、delivery_gate。

stage3_sync_payload：回写第三阶段的同步对象。含 topic_id、script_id、script_status、used_at、script_version、user_feedback_summary、publish_status、performance_stub。

performance_stub：复盘占位对象。含 publish_status、publish_date、platform、performance_status、notes 等。

G01｜阶段初始化与输入装配（Stage Init & Input Assembly）

节点目标：建立统一上下文，装配来自前三阶段与资料库的核心输入，锁定主路线和本次交付层级。

触发条件：用户进入第四阶段，且选择了题目或给出了可转化为题目的表达。

主执行角色：控制层

失败 / 异常条件：缺失定位报告、缺失第三阶段选题信息、用户身份上下文不完整。

回退 / 后续动作：阻断后续；返回缺失对象清单；必要时要求补齐上游对象。

G02｜模式识别与请求规范化（Mode Recognition & Request Normalize）

节点目标：识别当前属于已选题直接生成、自由想法梳理、修改重写还是批量出稿模式，并规范化 request_context。

触发条件：G01 完成后触发。

主执行角色：控制层 + 生成智能体

失败 / 异常条件：用户表达混乱，无法判断是“已有题”还是“模糊想法”；交付档位未明。

回退 / 后续动作：执行最小必要澄清；不得展开超过 1–2 轮问询。

G03｜路线锁定与冲突检测（Route Lock & Conflict Check）

节点目标：读取前序阶段已锁定的 ToB / ToC 主路线，检查当前需求是否与既定路线冲突。

触发条件：G02 完成后触发。

主执行角色：控制层 + 质检智能体

失败 / 异常条件：当前需求与既定路线明显冲突；用户要求的表达方向与主路线矛盾。

回退 / 后续动作：输出 route_conflict_flag 或 reroute_review_needed；不得自行改路线。

G04｜记忆加载与优先级解析（Memory Load & Priority Resolution）

节点目标：按固定优先级装配写作约束与参考依据，避免“模板压过用户本人”。

触发条件：G03 通过后触发。

主执行角色：生成智能体

失败 / 异常条件：长期偏好与本次会话要求冲突；模板库、技巧库和定位报告给出矛盾信号。

回退 / 后续动作：以用户当前明确要求优先；记录 override_reason。

二、输入优先级、记忆机制与证据边界

第四阶段最容易出错的地方不是“不会写”，而是“取材顺序错了、记忆错了、证据边界丢了”。因此本文件把输入优先级、记忆写回条件和证据回溯能力写成硬规则。

三、四种运行模式与模式级规则

4.1 最小必要澄清

首版只确认五项：用途 / 目标 / 时长 / 平台 / 风格。

若用户已明确说过“固定下来”“以后默认这样”，则直接读取记忆，不重复追问。

澄清上限：若缺口不影响首版生成，不得连续追问超过 1–2 轮；先出骨架或初稿，再继续修改。

6. 闭环优先：任何生成结果都应可记录、可追溯、可回流、可复盘。

1.3 禁止事项

不得因为模板库好用，就让模板压过用户本人。

不得因为技巧库更“爆”，就让技巧压过既定定位和路线。

不得默认整篇重写，用户明确要求局部修改时，必须优先执行局部修改。

2.1 输入调用优先级

用户当前明确要求：当前用途、目标、时长、平台、风格、特殊禁忌。

用户长期默认要求 / 记忆偏好：稳定的人设语气、平台习惯、时长偏好、表达偏好。

用户资料库 / 原始资料库：人物身份、业务信息、故事案例、表达素材、背书证据。

第一阶段访谈报告：用户真实情况、表达风格、业务背景与限制。

第二阶段定位报告：主路线、定位、阶段目标、内容方向、策略边界。

第三阶段选题规划：topic_id、阶段、选题来源、当前可用池状态。

技巧库（ToB / ToC）：路线对应的专业写法和展开习惯。

模板库：爆款拆解库：爆款模板库 、 爆款文案库 、爆款开头库、爆款金句库 、 爆款选题库 、爆款标题库

用户拆解库：更像本人、更像行业、更像过往有效内容的参考对象。

通用默认写作规则：仅在上层信息不足时作为兜底。

2.2 记忆机制

长期偏好：适合长期沿用的人设语气、平台偏好、长度偏好、表达风格。

项目偏好：本项目当前路线、当前阶段目标、当前内容方向。

本次会话要求：仅对当前任务有效的特殊要求。

写回条件：不是所有反馈都能升级为长期记忆。只有长期稳定偏好才进入长期偏好；阶段性偏好进入项目偏好；一次性要求只留在本次会话。

2.3 证据边界与回溯能力

不得编造用户未提供、资料库未记录、前序阶段未确认的经历、案例、数据、客户反馈或身份背书。

建议为每条文案结果保留最小证据回溯能力：source_trace、evidence_refs、evidence_gap_flag、low_confidence_sections。

若证据不足，但仍可先出首版，则应采用降级写法或标注低置信度，不得写成确定事实。
