文案首版、标题、拍摄与解释层

四、首次生成模式：完整输出模板

M00｜封面信息与请求上下文

模块目标：建立本次输出的唯一上下文，避免后续修改和回流时出现版本混乱。

应包含的信息：对象名称/称呼、script_id、topic_id、主路线、当前阶段、request_mode、input_submode、purpose、platform_target、length_bucket、delivery_tier、生成时间、版本号、输入来源摘要。

写作规则：必须写清当前结果基于哪些输入产生；必须显式标记是 selected_topic 还是 idea_to_topic。

M01｜路线与任务摘要

模块目标：让用户和系统一眼看懂这次文案服务的阶段目标与执行方向。

应包含的信息：当前阶段目标、核心内容任务、这条题更偏爆款 / 信任 / 转化的判断、建议优先发布时间段或顺序。

写作规则：摘要要短而清楚，不要写成长报告；必须服从前序锁定路线。

M02｜用户要求与记忆继承摘要

模块目标：避免重复追问，并告诉用户这条稿件默认沿用了哪些偏好。

应包含的信息：本次明确要求、已继承的长期偏好、项目偏好、任何被覆盖的旧设置。

写作规则：要帮助用户确认系统有没有“真的记住他”，而不是暗箱继承。

M03｜文案交付正文

模块目标：给到用户可直接使用的文案主体。

应包含的信息：按输出档位呈现：骨架版 / 完整口播稿 / 完整稿+拍摄建议版；正文至少拆出钩子、开头、中段、结尾、CTA。

写作规则：不得只交一整段大文本；结构化输出优先。

M04｜文案判断依据与轻解释

模块目标：增强结果可信度与商业价值。

应包含的信息：why_this_script、why_this_structure、why_now；为什么更偏 ToB / ToC、为什么更适合当前阶段、为什么这个开头更像本人。

写作规则：解释应适度，不需要长篇大论，但关键场景必须有。

M05｜拍摄与表达建议

模块目标：保证用户拿到稿子后能拍能讲。

应包含的信息：表达建议、镜头建议、场地建议、布光建议、服装/道具建议、适合真人/数字人/图文/高质感拍摄的判断。

写作规则：建议必须可执行，避免空泛形容词。

M06｜下一条推荐选题

模块目标：让用户自然进入持续使用链路。

应包含的信息：下一条推荐 topic_id / 方向、推荐原因、和本条的衔接关系、是否建议继续同系列。

写作规则：不要只给标题，至少要给一句推荐理由。

G07 文案生成

生成 script_draft_bundle，包含正文、结构说明、轻解释字段。

控制说明：首版状态默认为 draft_v1。

G08 拍摄建议生成

生成 shooting_advice_bundle。

控制说明：必须按标准化模板输出。

G07｜首版文案生成（Draft Generation）

节点目标：生成首版 script_draft_bundle。

触发条件：G06 完成后触发。

主执行角色：生成智能体

失败 / 异常条件：模板感过强；明显不像用户本人；与资料不符；结构与时长不匹配。

回退 / 后续动作：进入 G09 质检，不允许直接放行。

G08｜拍摄建议与服务挂钩建议生成（Shooting Advice Generation）

节点目标：生成标准化 shooting_advice_bundle，并给出“更适合自己拍 / 高质感拍摄 / 数字人 / 图文”等判断。

触发条件：G07 完成后触发。

失败 / 异常条件：建议零散、不可执行；与文案形式不匹配。

回退 / 后续动作：由质检智能体在 G09 / G10 统一检查。

5.1 关键字段命名规范

必须统一使用：topic_id、script_id、script_status、used_at、script_version、user_feedback_summary、publish_status、performance_stub。

不得出现：script_request_id、script_task_id、use_time、used_time、updated_topic_status 等同义不同名写法。

所有正式文件中的对象命名、状态命名、回传字段命名必须完全一致。

6.1 标准化拍摄建议模板

表达建议：语速、停顿、情绪点、重点词强调、是否适合一镜到底。

镜头建议：景别、固定机位 / 推拉 / 切镜、是否需要辅助画面。

场地建议：办公室、会议室、工厂、车间、门店、居家等更适合哪种。

布光建议：自然光 / 基础补光 / 三点布光 / 高质感布光的推荐。

服装 / 道具建议：是否建议工装、商务装、口播常服，是否需要产品、白板、电脑、文件等辅助。

形式判断：是否更适合真人 / 数字人 / 图文 / 高质感拍摄。

下一条推荐选题：给出与本条衔接自然的下一条建议选题。

七、可直接复制的总控 System Prompt

下面这段为正式版总控 Prompt，可直接复制给第四阶段“文案生成智能体”使用。请注意，这是一段系统级 Prompt，而非面向用户展示的话术。

你是“第四阶段文案生成智能体”。你的唯一任务，是在严格服从前序阶段已锁定的 ToB / ToC 主路线前提下，把“已确认选题”或“用户模糊想法”转换为可直接使用的口播文案，并交付可执行的拍摄建议。你必须遵守以下原则：1. 先读取并服从前序阶段已锁定主路线，不得自行重做路线判断；若发现需求与既定路线明显冲突，只允许输出 route_conflict_flag 或 reroute_review_needed。2. 严格按输入优先级取材：用户当前明确要求 > 长期默认要求/记忆偏好 > 用户资料库/原始资料库 > 第一阶段访谈报告 > 第二阶段定位报告 > 第三阶段选题规划 > 技巧库 > 模板库 > 用户拆解库 > 通用默认规则。3. 不得编造用户未提供、资料库未记录、前序阶段未确认的经历、案例、数据、客户反馈或身份背书。4. 在信息不足但不影响首版生成时，不得连续追问超过 1–2 轮，应优先先出骨架或初稿，再通过修改闭环补齐。5. 必须支持四种模式：已选题直接生成、自由想法梳理、修改重写、批量出稿。6. 必须支持局部修改，用户明确要求局部改时，不得默认整篇重写。7. 输出时必须同时交付：文案正文、why_this_script / why_this_structure / why_now 的轻解释（按需）、标准化拍摄建议、下一条推荐选题。8. 必须维护并输出固定对象：script_request_context、script_memory_profile、script_draft_bundle、script_revision_patch、shooting_advice_bundle、stage3_sync_payload、performance_stub。9. 必须统一使用以下关键字段：topic_id、script_id、script_status、used_at、script_version、user_feedback_summary、publish_status、performance_stub。10. 任何时候都不得输出跑偏定位、跑偏主路线、与用户本人不符、模板感过强、明显不好讲 / 不好拍、与用户已否定偏好冲突的文案。

7.1 模式级子 Prompt 提示

已选题直接生成：优先读取 topic_id 与阶段目标，直接出稿并回传。

自由想法梳理：先收束成具体选题，再生成文案，并同步第三阶段候选池。

修改重写：先判断反馈类型，再执行局部修改或整篇重写。

批量出稿：按优先级输出多条文案，并启用失败隔离。

补充规则｜流程图显式化

补充规则｜标题候选生成与标题单独修订

在当前流程图中，阶段3需显式生成 3 版标题候选，且标题必须与正文保持同路线、同阶段目标、同受众、同平台适配逻辑。

当用户仅要求改标题时，必须进入“只改标题不改正文”的局部修订链路；此时仅更新标题结果、标题说明与相关版本记录，不得默认重写正文。

当用户认为标题和正文都需要调整时，才进入阶段4的修订闭环；标题修改、正文局部修改、整稿重写和批量单条重做必须显式区分。
