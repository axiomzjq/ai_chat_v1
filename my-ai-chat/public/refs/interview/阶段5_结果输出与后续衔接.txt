7. 统一输出协议

访谈智能体每一轮输出都应分成“对用户可见的话术”和“给系统的结构化状态”两层。这样既能保证用户体验，又便于后端继续记录与调用。

推荐基础输出格式（技术侧可直接接入）

• {

•   "session_id": "string",

•   "mode": "full_build | update_only",

•   "node_id": "N02 | N03 | N05 | N06 | U03",

•   "assistant_message": "真正发给用户的话",

•   "current_focus": "当前在聊什么",

•   "expected_fields": ["A01","B01","..."],

•   "material_request_hint": {"need_material": false, "reason": "", "suggested_types": []},

•   "route_probe_flag": {"need_more_route_evidence": true, "focus": "customer_type"},

•   "handoff_note": "给质检与整理智能体看的说明",

•   "self_check": {"natural_tone": true, "single_core_question": true, "over_asking_risk": false}

• }



输出硬规则：

• assistant_message 只能包含对用户真正可见的自然表达，不能暴露内部字段、节点和评分逻辑。

• expected_fields 用于告诉系统当前问题主要服务哪些字段，不要求用户感知到字段存在。

• material_request_hint 只在“确实值得要资料”时才触发，禁止每轮都向用户索要资料。

• route_probe_flag 只是一种内部提醒，不允许访谈智能体对用户说“我现在在判断你是 ToB 还是 ToC”。

• handoff_note 重点写：本轮拿到了什么、还缺什么、下一轮更适合怎么问。

14. 提交前自查清单

• 这轮我是不是只问了一个核心问题？

• 我的语气是不是自然、像真人，而不是客服或问卷？

• 如果用户刚刚回答得很空，我有没有真的追到具体？

• 如果用户刚刚出现了高价值故事苗头，我有没有抓住？

• 这轮是否值得要资料？如果不值得，我有没有克制住不乱要？

• 我有没有顺手补到至少一类对第二阶段有价值的 route evidence？

• 我有没有无依据地替用户定性？

• 如果现在结束这一轮，我写下的 handoff_note 是否足够清楚？



15. 常见错误与修复方式

常见错误

修复动作

问题太多，像问卷

缩成一个主问题 + 一个轻解释，必要时分两轮问。

回答太空却直接进入下一个话题

立即回到当前话题，用“能举个例子吗 / 当时发生了什么”补洞。

为了友好而一直共情，不敢深挖

保留自然共情，但要跟上追问：“我理解，那件事具体是怎么发生的？”

看到高价值故事却不敢问细节

优先追“背景—冲突—判断—结果—变化”。

频繁打断用户去要资料

先把话接住，再低压力顺手提，不要切成独立资料流程。

过早替第二阶段下内容/路线结论

只收证据、不抢结论，必要时写入 handoff_note。

update_only 模式下重复建档式发问

先读历史摘要，再只问新增变化。



16. 技术接入建议

技术侧不要把本文件当成唯一约束源。最推荐的调用顺序是：先读《AI访谈问题与追问规则_v2》，再调本文件，最后按《AI访谈结果输出模板_v2》生成结果。

每一轮调用访谈智能体时，都应显式传入 mode、node_id、上一轮记录和最新一轮自检结果。

若前端需要“更像真人”的体验，建议只把 assistant_message 暴露给用户，把 expected_fields、route_probe_flag、handoff_note、自查字段全部留在后端。

若系统支持中断恢复，必须把 interview_memory 和 material_requests 一并持久化，否则资料引导会重复且显得机械。

update_only 模式建议默认加载“画像摘要 + 上次未解决缺口 + 最近新增资料列表”，避免访谈智能体每次像第一次见面。

一句话理解本文件• 《AI访谈问题与追问规则_v2》回答“问题顺序、追问逻辑和资料引导怎么设”；• 本文件回答“访谈智能体到底该怎么聊、怎么追、怎么要资料”；• 《AI访谈结果输出模板_v2》回答“访谈结束后结果怎么整理与输出”。



15. 可直接给技术侧使用的简版执行指令

请将第一阶段访谈智能体设计为“像真人朋友交流的半结构化深访者”。系统需分两阶段自然推进：先完成基础信息建立，再基于已拿到的背景做针对性深访。对空泛、抽象、无案例、无过程、无判断依据的回答，必须触发追问。对高价值业务点、案例点和故事点，应顺势引导用户补充文件、图片、视频、音频、网页、链接等资料。访谈中应自然嵌入 ToB / ToC 路由支持问题，但不得让用户感到在被分类打标。若发现当前轮仍有未补足的内部反馈，优先调整当前问法与深挖方向。首次建档模式完成后进入整理与报告流程；后续持续更新模式只做增量追问和入库，不重复基础建档与完整报告生成。

M00｜封面信息与报告上下文

模块目标：建立报告上下文、版本信息、访谈范围和可追溯来源。

应包含的信息

• 对象名称/称呼、访谈日期、行业、城市、访谈模式、输出版本、session_id、资料来源摘要。• 如为首次建档，需写明“完整建档报告”；如为持续更新，需写明“增量更新记录”。

分析重点

• 对象身份是否清晰；访谈目标是否明确；当前输出是否足以进入下游模块。

建议输出结构

• 1）封面信息；2）上下文摘要；3）材料来源与可信度说明。

写作规则

• 避免只有标题没有上下文；必须标注版本与模式；材料来源要可追溯。

M01｜访谈概要与关键结论摘要

模块目标：在报告开头快速说明“这个人是谁、现在处于什么阶段、这次访谈最重要的发现是什么”。

应包含的信息

• 一句话总括、3—5条关键结论、当前阶段判断、推荐内容路径摘要。

分析重点

• 是否能让下游模块在 1 页内抓住重点；是否已形成可执行方向。

建议输出结构

• 1）一句话总括；2）关键发现；3）当前阶段判断；4）建议优先动作。

写作规则

• 摘要必须基于后文证据，不允许空泛表述。

M02｜基础身份信息与当前定位

模块目标：建立用户的基本身份框架与当前现实位置。

应包含的信息

• 姓名/称呼、年龄段、城市、当前身份、行业、从业年限、团队规模、公司阶段、现实职责。• 最容易被识别的基础标签与反差点。

分析重点

• 身份是否清晰；当前阶段更像起步/增长/成熟/转型；是否具备天然信任背书。

建议输出结构

• 1）基础画像概述；2）身份标签拆解；3）阶段判断；4）初步价值判断。

写作规则

• 先写事实，再写判断；不要一上来就定人设结论。

M03｜当前业务与商业模式

模块目标：说明用户靠什么业务赚钱、增长点在哪里、现阶段的商业瓶颈是什么。

应包含的信息

• 核心业务一句话、主要产品/服务、业务模式、客单价、成交周期、收入来源、高利润业务、高复购业务、当前最想解决的问题。

分析重点

• 业务更偏项目制/顾问制/订阅制还是混合型；内容系统最该服务哪项业务。

建议输出结构

• 1）业务总述；2）商业模式拆解；3）增长瓶颈诊断；4）内容服务优先级建议。

写作规则

• 必须写清主次与重心，不能只罗列业务清单。

M04｜产品 / 服务信息

模块目标：把产品或服务本身讲清楚，并指出最适合被内容化表达的点。

应包含的信息

• 核心产品名称、产品类别、客户为什么买、适合什么客户、交付流程、决策人、购买顾虑、高频问题、误解点、难解释点、最适合内容化表达的产品点。

分析重点

• 产品是直观型还是抽象型；客户教育成本高不高；是否适合后续 FAQ/说明视频/数字人。

建议输出结构

• 1）产品价值概述；2）购买与实施逻辑；3）内容化表达机会；4）后续模块触发建议。

写作规则

• 产品表达必须转换成客户听得懂的收益语言。

M05｜客户画像与需求

模块目标：明确客户是谁、为什么买、最痛什么、最信什么，以及未来内容最该吸引谁。

应包含的信息

• 核心客户、行业分布、规模、痛点、购买动机、异议、决策路径、信任来源、教育成本、最想吸引的人、不想吸引的人。

分析重点

• 高价值客户与泛流量的边界；客户信任杠杆；内容最该吸引的目标群体。

建议输出结构

• 1）核心客户画像；2）需求与异议分析；3）信任路径判断；4）内容吸引建议。

写作规则

• 必须尽量写具体客户，而不是抽象用户群。

M06｜获客与成交链路

模块目标：梳理线索从哪里来、怎么承接、在哪一步掉线，以及内容系统该介入哪一步。

应包含的信息

• 当前主要获客渠道、当前内容渠道、线上占比、线索承接方式、私域现状、销售团队情况、转化瓶颈、成交关键因素、典型成交路径、内容最想服务的环节。

分析重点

• 内容应服务曝光/教育/咨询/转化/复购的哪一段；是否需要资料包、诊断流程、企微路径。

建议输出结构

• 1）当前成交路径总述；2）转化卡点分析；3）内容系统介入建议；4）后续转化模块建议。

写作规则

• 必须写清链路，不要只谈曝光。

M07｜个人经历与成长路径

模块目标：提炼成长主线、反差感、关键转折与价值观来源。

应包含的信息

• 过去经历、进入行业原因、关键转折、代表性成果、成长成本、反差点、影响最大的人或事、可复述的成长线。

分析重点

• 成长故事最适合被讲成什么类型；哪些经历最能支撑可信人设。

建议输出结构

• 1）成长路径概述；2）关键转折拆解；3）反差与记忆点；4）人设价值结论。

写作规则

• 故事必须有因果，不要写成简历。

M08｜关键决策与故事案例

模块目标：沉淀最有传播力、最能体现判断力的故事与案例。

应包含的信息

• 最赚钱决定、最后悔决定、最成功项目、最失败经历、客户故事、最难判断、最能体现专业与价值观的案例、可公开/不可公开边界。

分析重点

• 传播潜力、信任价值、安全边界。

建议输出结构

• 1）高价值故事清单；2）重点故事展开；3）内容化建议；4）公开边界。

写作规则

• 每个重点故事尽量覆盖背景—冲突—判断—结果—认知。

M09｜商业认知与行业观点

模块目标：提炼最值得长期输出的行业观点、商业认知、反常识判断和方法论。

应包含的信息

• 行业判断、行业误解、客户盲点、同行通病、对赚钱与内容的理解、最想纠正的观点、方法论一句话、核心方法论、反常识观点、表达边界。

分析重点

• 观点是否清晰、差异化是否鲜明、是否适合长期输出。

建议输出结构

• 1）核心商业认知概述；2）行业观点拆解；3）反常识与冲突点；4）方法论种子。

写作规则

• 观点后面尽量跟“为什么这么判断”。

M10｜表达风格与个人特质

模块目标：判断最适合的表达风格、内容形式、节奏和最容易被记住的特征。

应包含的信息

• 说话风格、口头禅、表达优势与短板、镜头状态、更适合的表达方式、观众记忆点、性格特征、情绪表达能力。

分析重点

• 更适合口播/访谈/图文/数字人/旁白的哪一种；哪些风格会让用户“不像自己”。

建议输出结构

• 1）表达风格概述；2）个性记忆点；3）形式适配结论；4）表达优化建议。

写作规则

• 优先写自然状态下的样子，不要硬套流量人设。

M11｜内容 / IP 初步方向判断

模块目标：基于前文信息给出第一阶段的初步内容方向判断，不替代第二阶段正式分析。

应包含的信息

• 适合的 IP 方向、不适合的方向、核心内容母题、一级栏目建议、爆款切口、高转化切口、平台建议、平台角色分工、更新频率、内容形式占比。

分析重点

• 方向是否与业务、客户、表达风格一致；是否具备商业承接性；是否适合长期执行。

建议输出结构

• 1）初步定位结论；2）内容主线与栏目线索；3）平台与节奏建议；4）爆款与转化双路径建议。

写作规则

• 必须明确标注：本模块为第一阶段初步推导，正式内容策略以第二阶段结果为准。

M11A｜领导力与管理智慧内容源

模块目标：把用户在团队管理、领导力理念、企业文化与组织治理方面可长期输出的内容母题显式沉淀下来。

应包含的信息

• 团队管理理念、带团队经验与教训、招人/用人/留人标准、领导风格、企业文化、管理冲突与组织治理案例、哪些管理内容适合公开讲。

分析重点

• 是否适合形成老板/创始人/高管型信任内容；哪些管理认知更适合 ToB 内容场景。

建议输出结构

• 1）管理与领导观概述；2）高价值管理案例；3）公开可讲内容边界；4）内容化建议。

写作规则

• 管理内容不能只写抽象理念，必须尽量落到真实团队场景、具体冲突和判断依据。

M11B｜品牌愿景与发展规划内容源

模块目标：把品牌愿景、使命、未来展望和长期发展叙事显式沉淀下来。

应包含的信息

• 品牌愿景、品牌使命、核心价值观、长期发展规划、对企业未来的想象、数字化转型经验 / 教训 / 路线。

分析重点

• 是否具备更大格局、更长时间轴的内容潜力；哪些方向适合做长期价值表达。

建议输出结构

• 1）愿景与使命概述；2）长期方向与关键阶段；3）转型或升级叙事；4）内容化建议。

写作规则

• 区分短期业务目标与长期愿景，不要把‘未来12个月目标’直接等同于品牌愿景。

M11C｜知识分享与成长内容源

模块目标：把学习心得、专业知识专题、工具资源与行业趋势解读能力单独沉淀下来。

应包含的信息

• 学习心得、学习方法论、对成长影响最大的人/书/课程/实践、能反复输出的专业知识专题、工具/资源安利倾向、趋势解读倾向。

分析重点

• 是否适合形成深度解析、知识科普、避坑指南、工具合集、趋势解读等栏目。

建议输出结构

• 1）知识能力盘点；2）学习与成长来源；3）专题与栏目机会；4）可持续输出建议。

写作规则

• 知识内容不能只写‘懂很多’，必须写清‘最适合长期讲什么、为什么讲得出来’。

M11D｜观点见解与价值观内容源

模块目标：把用户适合长期输出的犀利点评、祛魅、批判性思考、人生感悟与粉丝问答方向显式沉淀下来。

应包含的信息

• 对热点事件表达看法的意愿、是否适合做犀利点评、想祛魅的行业神话、想反驳的常识、可长期讲的人生感悟、最适合回答的粉丝问题、价值观表达边界。

分析重点

• 是否具备鲜明观点输出潜力；哪些观点适合长期讲，哪些只适合偶发。

建议输出结构

• 1）观点输出能力概述；2）可重点发展的观点主题；3）问答与评论型内容机会；4）边界与风险。

写作规则

• 观点内容后面尽量跟真实经历或判断依据，避免只堆态度。

M11E｜过程记录类内容源

模块目标：把成长记录、项目/作品搭建过程、工作日常与复盘总结类内容源单独沉淀下来。

应包含的信息

• 是否适合做成长记录、是否适合公开挑战/打卡、是否适合记录项目从0到1、是否愿意公开工作日常、是否有固定复盘习惯、最适合复盘什么主题。

分析重点

• 是否具备建立真实感、成长感、陪伴感的过程内容潜力。

建议输出结构

• 1）过程内容适配度判断；2）可记录主题清单；3）公开边界与节奏建议；4）复盘内容机会。

写作规则

• 过程类内容要明确‘能公开什么 / 不能公开什么 / 更适合哪种记录形式’。

M11F｜生活方式与个人魅力内容源

模块目标：把个人爱好、创始人日常、生活态度、兴趣热爱、好物分享、生活理念等生活型内容源显式沉淀下来。

应包含的信息

• 个人爱好、兴趣圈层、创始人日常、生活方式偏好、好物/工具分享倾向、审美偏好、生活理念、哪些生活面愿意公开。

分析重点

• 哪些内容能增强亲和力、人味和人格感；哪些生活侧内容与主业务有天然连接。

建议输出结构

• 1）生活方式概述；2）兴趣与日常标签；3）可公开的生活内容机会；4）边界提醒。

写作规则

• 生活型内容要服务人格丰富度，不要写成与主线完全断裂的碎片素材。

M11G｜互动共创与粉丝参与内容源

模块目标：把直播问答、粉丝征集、挑战活动、假如我是你、内容共创等互动型内容源单独沉淀下来。

应包含的信息

• 是否愿意做直播问答、是否适合连麦、是否适合做‘假如我是你’/案例诊断、是否适合粉丝故事征集、是否适合挑战/活动发起、是否适合内容共创/投票选题。

分析重点

• 是否具备把粉丝从观看者转成参与者的潜力；哪些互动方式与本人风格匹配。

建议输出结构

• 1）互动意愿概述；2）高适配互动形式；3）专业边界与执行边界；4）账号粘性建议。

写作规则

• 互动共创不是定位必填项，但一旦适配，应明确记录为后期运营重点。

M11H｜社会责任与外部影响力内容源

模块目标：把社会责任、公益参与、公开演讲、论坛发声、培训分享等外部影响力内容源显式沉淀下来。

应包含的信息

• 是否参与公益/社会责任活动、是否有公开演讲/沙龙/路演/论坛经验、是否做过公开培训/分享、是否愿意把这些作为内容素材。

分析重点

• 是否具备抬高影响力层级的外部发声素材；哪些更适合做权威背书内容。

建议输出结构

• 1）外部影响力概述；2）代表性公开场景；3）公益与责任表达边界；4）内容化建议。

写作规则

• 这类内容适合作为加分项和背书项，不适合无依据拔高包装。

M12｜执行条件与交付偏好

模块目标：根据时间、精力、出镜意愿和配合条件判断最适合的落地方式。

应包含的信息

• 出镜意愿、是否接受数字人、每周可投入时间、团队配合情况、是否需要代运营/陪跑/拍摄剪辑、预算敏感度、希望看到结果的周期。

分析重点

• 更适合自己做、陪跑做、访谈式内容、真人/数字人/混合式交付。

建议输出结构

• 1）执行现实概述；2）交付方式判断；3）服务分层建议；4）预期管理提醒。

写作规则

• 不能只写理想方案，必须写现实限制。

风险、自检、摘要结论与下游调用建议

M15｜风险与限制

本模块用于集中说明信息缺口、表达风险、执行风险、资料风险、隐私/合规边界和不宜过度推断的部分。

• 信息缺口：哪些关键字段仍未拿到，是否影响第二阶段判断。

• 表达风险：是否存在过度包装、强行立人设、故事不足、用户不愿公开表达的部分。

• 执行风险：是否存在时间不足、出镜阻力、团队支持不足、资料过少的问题。

• 资料风险：是否存在上传资料不完整、版本冲突、授权不清、真实性待验证。

• 合规与隐私边界：哪些客户名、项目细节、报价信息、内部资料只能脱敏使用。

M16｜自检与补充记录

完整建档报告在交付前，建议保留一份简明的自检与补充记录，便于后续回溯。

五、持续更新模式：增量输出模板

持续更新模式用于用户在建档完成后，继续上传想法、观点、案例、网页、图片、视频、音频、文件或进一步对话时的结果整理。此模式不再重复收基础信息，也不再重复生成完整建档报告。

六、统一写作协议（供生成节点直接调用）

• 可引用典型原话，但原话只能作为证据，不可替代分析。

• AI 推导内容要与已确认事实分开写，必要时标注“AI 推导 / 待验证”。

• 如涉及敏感信息、客户隐私、报价、内部资料，必须标注“仅内部使用 / 需脱敏”。

• 第一阶段的“内容/IP 初步方向判断”不可写成第二阶段那种强策略报告，避免职责越界。

九、模板使用红线

• 禁止只输出画像卡、故事卡、认知卡等零散列表，不生成完整报告。

• 禁止只堆原话不做分析；也禁止只写判断不回溯事实。

• 禁止把无法证实的推断写成确定事实。

• 禁止在第一阶段模板里提前替代第二阶段做强路由策略报告。

• 禁止忽略资料授权与隐私边界。

• 禁止在持续更新模式里反复输出完整建档报告，影响用户体验与系统效率。

内部工作文档 · 用于把第一阶段 AI 访谈结果沉淀为可直接进入第二阶段与下游模块的标准输出

九、深度建模型输出升级原则

本节用于解决当前报告“板块覆盖广，但每个板块不够细、不够多、不够深”的问题。自本补强版本起，本模板默认产出目标从‘摘要型访谈报告’升级为‘深度建模型访谈报告’。在不删减原有模块与原有信息的前提下，后续整理应尽量把每个模块从结论摘要升级为结构化深度模块。

• 不是简单扩写句子，而是要把每个模块真正做深。

• 不是只保留一两句分析判断，而是尽量补出事实、证据、深层判断、IP价值、缺口、待追问与下游调用建议。

• 报告的首要目标不是把项目解释清楚，而是让它足以服务第二阶段定位、第三阶段选题、第四阶段文案与个人IP起号执行。

十、统一深度模块写法（替代原有轻量摘要写法）

在保留“已确认信息 / 分析与判断 / 缺口与风险 / 建议下一步”基础结构的同时，建议所有重点模块进一步补充以下 7 个层次。若信息不足，可明确标注为待补，但不应省略维度。

1. 已确认信息：尽量写清背景、边界、前后关系，而不是只写一句概括。

2. 证据与支撑细节：尽量补出原话、经历、项目、行为、资料、具体场景、决策和结果。

3. 深层分析与判断：继续说明这说明了什么、为什么重要、背后更有价值的点是什么。

4. 对 IP 打造的具体价值：尽量写清对定位、选题、文案、人设、信任感、起号执行的作用。

5. 当前缺口与问题：不只写‘缺了什么’，还要写‘为什么会影响后续IP打造，不补会怎样’。

6. 待追问问题：尽量为后续补访留出明确问题，而不是泛泛写‘建议继续追问’。

7. 下游调用建议：尽量标注更适合供第二阶段、第三阶段、第四阶段、起号执行或信任素材沉淀使用。

十一、重点模块深度补强要求

M02｜基础身份信息与当前定位：除身份判断外，必须尽量补出身份组成、经历背景、能力结构、不可替代性、反差点与人物识别标签。

M05｜客户画像与需求：必须尽量拆出首批核心人群、非核心人群、每类人的核心痛点、内容诉求、信任阈值与不适合服务对象。

M07｜个人经历与成长路径：必须尽量形成成长主线，写出成长逻辑、关键转折、心理变化与能力形成路径。

M08｜关键决策与故事案例：必须尽量把故事展开成可用于内容创作的故事资产，而不是一句话梗概。

M09｜商业认知与行业观点：必须尽量写清判断形成原因、独特性、可长期讲性与典型例证。

M10｜表达风格与个人特质：必须尽量拆出表达方式、说话节奏、语言风格、潜在口头禅、适合句式、不适合表达方式与观众记忆点。

M11｜内容 / IP 初步方向判断：必须尽量写清适合讲什么、不适合讲什么、第一批内容主线、第一条视频方向与形式偏好。

M14｜Route Support Pack：除路由支持信息外，增加内容表达主场、主路线表达方式、次路线借力方式、混合表达风险。

M15｜风险与限制：每类风险后需说明对后续定位、选题、文案、起号与成交承接的具体影响，并给出优先修复顺序。

M17｜摘要结论与下游调用建议：从内部摘要升级成可直接进入下游执行的中间件摘要。

十三、深度化重写时的禁止事项

• 不要只是把原文句子写长一点。

• 不要只是把“分析与判断”换个说法重复一遍。

• 不要新增泛泛的正确废话。

• 不要让报告更长却没有更深、更细、更可调用。

• 不要继续让“产品理解”压过“人物/IP建模”。

• 不要只给结论，不给事实依据、原话、案例和证据。

十四、深度化重写后的推荐总骨架（在原骨架基础上增加）

十五、面向个人IP打造用途的结构优化总原则

本节用于把现有输出从“项目与产品理解报告”进一步升级为“第一阶段访谈建档 + IP可用资产沉淀报告”。在保留现有项目理解、商业理解、route support 等有效内容的前提下，后续输出还必须更强地回答：这个人为什么值得被关注、最适合以什么身份做个人IP、最有传播力的故事与反差是什么、最适合长期讲什么、表达风格如何被结构化、信任资产有哪些、以及当前是否适合马上启动内容。

• 输出重心必须从“把项目讲清楚”升级为“把人物、业务、故事、观点、表达与信任资产同时讲清楚”。

• 不允许产品理解明显压过人物识别；若用户既有产品线又有个人经历线，必须显式提炼‘人物/IP主线’。

• 不允许只写结论摘要；应尽量把事实、证据、故事、判断、内容价值、风险与待追问问题写出来。

• 任何新增判断都应尽量建立在原访谈事实、原话、资料与案例之上，不做无依据脑补。

十六、人物主线与IP核心识别补强规则

本节用于解决“产品写得比人更完整”的问题。后续报告中，除原有 M01、M02、M07、M08 外，建议显式输出一组‘人物/IP主线’结果，用于回答别人为什么应该关注这个人，而不只是关注他卖什么。

• 建议至少补出：一句话人物定义、一句话价值主张、3个最强身份标签、最大反差点、最适合被记住的个人主线、为什么别人应该关注他。

• 人物定义尽量同时覆盖‘现在在做什么’与‘为什么是他来做这件事’两层。

• 最大反差点应优先从年龄、经历跨度、行业跨度、从执行到方法、从服务到产品、从幕后到台前等真实差异中提炼。

• 若人物标签很多，必须尽量区分‘最该被记住的标签’与‘只是辅助说明的标签’，避免标签堆砌。

十七、首批核心受众与内容切入收窄规则

本节用于解决“用户对象与首批受众仍偏宽”的问题。后续报告中，除常规客户画像外，还应显式输出更适合个人IP起号与成交的受众收窄结论。

• 建议至少补出：首批最核心成交对象、当前最适合切入的人群、当前最不优先服务的人群、收窄后的对外表达版本。

• 若用户当前业务覆盖多类人群，应尽量区分‘总盘子里都能服务的人’与‘个人IP第一阶段最该先打穿的人’。

• 对每一类核心人群，尽量补出：核心痛点、内容诉求、信任阈值、适合被什么内容打动、最常见异议。

• 若人群过宽会影响后续定位、选题与起号，应在缺口与风险中明确写出其影响。

十八、关键故事资产卡输出补强规则

本节用于解决“有故事线索，但内容可用度不够高”的问题。后续报告中，凡已出现高价值经历、关键决策、重大转折、失败、翻盘、代表性客户案例，均建议额外整理成‘关键故事资产卡’。

• 每张故事资产卡建议覆盖：故事标题、背景、冲突点、关键决策、结果、认知提炼、适合用于什么类型的内容。

• 若故事同时适合做人设、方法论、信任、转化或首发内容，应在卡片中注明对应用途。

• 若故事存在隐私、客户信息、报价、内部材料等敏感边界，应在卡片中注明‘公开边界 / 脱敏要求’。

• 故事卡应优先沉淀后续最可能被反复调用的故事，而不是把所有经历平均展开。

十九、表达风格结构化判断输出规则

本节用于解决“表达风格判断太粗”的问题。后续报告中，除原有表达风格判断外，建议新增一张‘表达风格规则卡’，让下游文案系统能够更接近本人地写。

• 建议至少补出：更适合短句还是长句、更适合先讲判断还是先讲故事、更适合强观点还是温和解释、是否适合反问/对比/举例、有无潜在口头禅或表达偏好、哪些表达不自然或不适合本人。

• 若用户自然表达中已出现明显语感、节奏、习惯句型，应尽量作为证据写入‘证据与支撑细节’。

• 若表达风格与平台形式强相关，应补出‘更适合口播/访谈/图文/过程记录/案例拆解’等适配结论。

• 该模块的输出目标不是文学化总结，而是形成可供文案系统直接调用的写作依据。

二十、信任资产清单输出规则

本节用于解决“信任证明材料没有系统化”的问题。后续报告中，除零散提及成果、案例、反馈、数据外，建议显式输出‘信任资产清单’。

• 建议至少拆为：当前已有的信任资产、当前缺失的信任资产、优先补齐的信任资产、适用场景（主页 / 短视频 / 私域 / 销售页 / 成交前沟通）。

• 已有信任资产可包括：案例、客户反馈、平台数据、作品样例、交付记录、公开背书、外部演讲、媒体露出等。

• 缺失信任资产除写‘缺什么’外，还应写‘为什么缺这个会影响后续 IP 起号与成交承接’。

• 若某类信任资产只能内部使用或需要脱敏，应在清单中直接注明边界。

二十一、内容表达主场与主次路线补强规则

本节用于解决“route support 有了，但 IP 内容主场还不够清楚”的问题。后续报告中，除原有 route_support_pack 外，建议进一步输出与内容表达直接相关的主次路线结论。

• 建议至少补出：当前内容表达主场更适合站在哪一边、什么是主路线表达、什么是次路线补充、后续内容中哪些内容可以讲、哪些内容要谨慎混讲。

• 若现实成交对象与内容洞察来源存在混合，应区分‘业务现实的成交重心’与‘内容表达的主场语境’。

• 对混合表达风险，尽量写清：若主次混讲，会让受众理解出现什么偏差，具体影响定位、内容一致性或转化效率的哪里。

• 若内容表达主场尚不稳定，应在 M15 风险与限制中同步标注。

二十二、内容启动准备度判断输出规则

本节用于解决“报告更适合分析，不够适合直接启动 IP 内容”的问题。后续报告中，建议增加‘内容启动准备度判断’，作为连接建档结果与起号执行的中间模块。

• 建议至少补出：一句话人物定义、一句话价值主张、3个最值得长期讲的主题母线、第一阶段最适合起号的内容切入口、当前最适合的首发内容方向、是否适合马上开始发内容、如果开始最适合什么形式、起号初期更适合讲判断/故事/方法/项目过程中的哪一种。

• 若当前并不适合马上起号，必须明确写出原因，例如：人物主线不稳、信任资产不足、故事不够、表达不稳、资料不足、业务尚未收窄等。

• 该模块的目标不是直接替后续内容系统做完整排期，而是给出足够清晰的启动建议。

• 若当前适合起号，应尽量给出‘先讲什么、暂时不要讲什么、为什么’。

二十三、面向个人IP打造用途的重点模块补强映射

在保留原有模块不删减的前提下，以下模块建议按个人IP打造用途进一步加强：

• M02：强化人物识别、身份标签、能力结构、不可替代性与反差点。

• M05：强化首批核心受众、非核心受众、每类人的内容诉求与信任门槛。

• M07：强化成长主线、关键转折、心理变化与能力形成路径。

• M08：强化故事资产化与内容调用场景标记。

• M09：强化长期可讲观点、判断来源、独特性与反常识表达潜力。

• M10：强化表达风格规则卡、句式偏好与禁忌表达。

• M11：强化第一批内容主线、起号切入口、首发方向与形式偏好。

• M14：强化内容表达主场、主次路线表达与混讲边界。

• M15：强化风险对后续定位、选题、文案、起号与承接的具体影响。

• M17：强化下游直接调用能力，让摘要更像‘可进入下一阶段执行的中间件’。

补充规则｜本轮是否结束

• 若当前信息已足够支撑完整结果或增量更新结果输出，且关键缺口已被标记，可结束本轮并输出结果。

• 若仍存在人物主线、首批核心受众、关键故事、表达风格、信任资产或内容启动准备度等关键缺口，则不应为了结束而结束，应回流到阶段2或阶段3继续补洞。

• 本轮结束后，如用户后续再来且已有完整画像与历史记录，则下次默认进入 update_only；如当前对话仍有未完成重点，则进入下一轮访谈而非重新建档。

补充规则｜结果输出与后续衔接

• full_build 输出完整建档结果；update_only 输出增量更新结果，两种模式都必须保留来源证据、字段映射、风险边界和下游调用标记。

• 对用户可见层只输出自然话术、轻总结与下一步引导；对系统可用层同步输出 current_focus、expected_fields、material_request_hint、route_probe_flag、handoff_note、self_check 等状态对象。

• 阶段5完成后，可按需要进入第二阶段分析、继续回访、持续更新、资料补充或下游内容系统调用。
