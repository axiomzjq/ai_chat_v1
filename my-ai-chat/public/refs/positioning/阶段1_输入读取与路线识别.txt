阶段1_输入读取与路线识别

路线识别、表达主场、首批受众收窄与技巧库挂载规则

ToB / ToC 路线识别规则表

用于第二环节 IP 分析系统的路由识别与单智能体生成前置规范

使用说明：本规则表用于在生成 IP 分析报告前，先自动判断用户属于 ToB 创始人 IP 还是 ToC 个人 IP，并将判定结果作为后续所有模块的前置路由条件。



一、文档目标

本规则表用于在第二环节正式生成 IP 分析报告之前，先自动判断用户属于哪一条主路线：ToB 创始人 IP 或 ToC 个人 IP。

判定结果将直接作为后续全部模块的前置路由条件，确保账号定位、内容规划、平台建议、转化建议不跑偏。

系统需能够处理典型 ToB、典型 ToC 与混合型用户，但正式生成时只允许走一条主路线。

二、适用范围

适用于所有已完成第一环节 AI 访谈与信息收集，并产出结构化访谈报告的用户。

典型对象包括：创始人、联合创始人、企业负责人、顾问/咨询师、培训老师、个人品牌创业者、高客单服务从业者、面向个人消费者的内容型用户。

三、路线定义

ToB 创始人 IP 定义

符合以下核心特征中的多数：

产品或服务主要面向企业、工厂、政府、机构、组织。

成交对象不是普通个人，而是组织中的决策者/影响者。

决策链条较长，成交更依赖专业信任、案例证明、行业理解、服务逻辑和结果交付。

内容主要服务于建立信任、客户教育、咨询前筛选和成交前认知铺垫。

典型例子：

企业服务公司创始人

制造业老板

展厅设计施工公司负责人

系统集成/软件服务商

咨询顾问型 ToB 创业者

政企项目服务提供方

ToC 个人 IP 定义

符合以下核心特征中的多数：

产品或服务主要面向个人消费者。

成交对象是个人，而不是组织。

决策链较短，成交更依赖情绪共鸣、个人吸引力、生活方式认同、内容互动和体验感受。

内容主要服务于吸引关注、激发兴趣、建立喜好以及促进个人购买或加入社群。

典型例子：

个人成长博主

情感/生活方式博主

面向个人的咨询师/教练

课程知识付费型个人 IP

面向个人的服务型创作者

小众兴趣/技能型博主

四、核心判定维度

建议用 8 个维度综合判断，而不是只看“用户自称是 ToB 还是 ToC”。

维度1：目标客户类型

判断说明：最高优先级维度。判断产品/服务主要卖给谁，最终付钱的人是企业还是个人。

判定规则：明确面向企业/工厂/政府/机构 → ToB 强信号；明确面向个人消费者 → ToC 强信号。

维度2：主要收入来源

判断说明：看现在主要收入来自哪类客户，最大单笔收入通常来自企业还是个人。

判定规则：主要收入来自企业项目服务 → ToB；主要收入来自个人购买/课程/社群/个人服务 → ToC。

维度3：成交链路长度

判断说明：看从认识到成交通常要多久，是否需要多轮沟通、方案、比价与审批。

判定规则：较长决策链、多人参与 → ToB；个人快速决策、短链路 → ToC。

维度4：成交依赖因素

判断说明：看客户为什么买，是依赖专业信任，还是依赖个人吸引/情绪打动。

判定规则：依赖专业度、案例、交付能力、组织信任 → ToB；依赖个人魅力、内容吸引力、情绪/痛点共鸣 → ToC。

维度5：内容主要目的

判断说明：看内容主要服务什么结果。

判定规则：主要服务咨询、线索、成交前教育 → ToB；主要服务吸粉、互动、种草、个人转化 → ToC。

维度6：典型受众画像

判断说明：看受众描述的是岗位角色还是生活身份。

判定规则：受众以“老板/采购/品牌总监/行政负责人”等角色描述 → ToB；以“宝妈/大学生/女生/职场新人”等个人身份描述 → ToC。

维度7：内容表达风格

判断说明：看内容更适合讲行业认知与案例，还是故事、情绪与生活方式。

判定规则：偏行业观点、决策经验、案例拆解、客户教育 → ToB；偏故事表达、共鸣、个人成长、痛点转变 → ToC。

维度8：平台优先级倾向

判断说明：辅助维度，不单独决定路线。

判定规则：更适合视频号+公众号 → 偏 ToB；更适合抖音+小红书 → 偏 ToC。

六、判定打分机制

建议不是简单“像 ToB 就判 ToB”，而是为 ToB 与 ToC 两条路线分别打分。

1. ToB 分值项（每项 0–2 分，满分 16）

产品/服务面向企业或组织

主要收入来自企业客户

成交链较长

内容目标偏咨询 / 信任 / 教育

受众描述偏岗位角色

内容风格偏行业 / 案例 / 方法

依赖专业信任与结果

平台更偏视频号 / 公众号

2. ToC 分值项（每项 0–2 分，满分 16）

产品/服务面向个人

主要收入来自个人客户

成交链较短

内容目标偏流量 / 互动 / 种草

受众描述偏个人身份

内容风格偏故事 / 共鸣 / 生活方式

依赖个人吸引与情绪带动

平台更偏抖音 / 小红书

3. 最终判定规则

ToB 分 ≥ 11，且比 ToC 高 3 分以上 → 判定 ToB。

ToC 分 ≥ 11，且比 ToB 高 3 分以上 → 判定 ToC。

两边都高但分差 < 3 分 → 进入混合型处理。

两边分都低，或关键字段缺失过多 → 输出 need_more_info。

七、混合型处理规则

现实中会有同时具备 ToB 和 ToC 属性的用户，例如既卖企业咨询，也卖个人课程。

正式生成时只允许走一条主路线，避免整份报告混写。

主路线优先级依次看：主要收入来源 → 最想吸引的客户 → 内容最终要服务的成交对象。

系统可保留 secondary_route_type 作为备注，但不进入主体生成逻辑。

八、路由后的生成原则

若判定为 ToB，后续生成必须强调：

定位更偏专业、行业、信任、价值。

核心受众更偏岗位 / 决策者。

高传播切口与高转化切口严格区分。

平台优先级更偏视频号、公众号。

私域承接更偏微信 / 企微 / 资料包 / 咨询。

选题更偏行业误区、案例拆解、决策认知、客户教育、方法逻辑。

若判定为 ToC，后续生成必须强调：

定位更偏个人特色、用户价值、记忆点。

核心受众更偏个人身份画像。

内容更强调痛点、情绪、共鸣、转变。

平台优先级更偏抖音、小红书。

私域承接更偏私信关键词、社群、清单、小课。

选题更偏情绪共鸣、个人故事、用户痛点、前后变化、结果展示。

九、生成流程中的位置

生成前准备：检查第一环节结构化输入是否完整。

步骤1：路线识别 — 由生成智能体先输出 route_type、route_confidence、route_evidence。

步骤2：路线一致性自查 — 检查路由是否清晰、是否需要重判。

只有路线一致性自查通过后，才能进入正式模块生成。

十一、路由识别后的自查项

从主体模块开始，生成智能体在每个模块都要增加“路线匹配度自查”。

要判断当前模块内容是否符合判定路线，是否出现 ToB / ToC 混写。

一旦发现平台建议、账号风格、选题方向与路线不一致，必须直接重写相关模块，而不是只做轻量措辞优化。

十二、常见错误判定案例

错误1：明明是 ToB，却写成泛个人博主

表现：账号简介太网感、选题全是情绪/成长鸡汤、平台建议全压抖音/小红书、没有咨询前教育与案例逻辑。

处理：重写相关模块，强制恢复 ToB 逻辑。

错误2：明明是 ToC，却写成企业顾问口吻

表现：定位太重专业术语、选题过于案例化/行业化、平台建议不考虑抖音/小红书、私域承接过于企微/咨询。

处理：重写相关模块，强制恢复 ToC 逻辑。

错误3：混合型用户报告混写

表现：前文像 ToB，后文像 ToC；平台建议与选题风格冲突；核心受众一会儿是老板，一会儿是普通用户。

处理：回到路线识别步骤重判主路线，并在指定主路线后重新生成。

十三、面向个人IP内容主场的补充判定

路线识别的结果，不仅用于决定报告写成 ToB 还是 ToC，还必须进一步服务“这个人后续应站在哪个内容表达主场说话”。因此，在 route_type 判定通过后，建议继续补充“内容表达主场”“主路线表达方式”“次路线补充方式”“混合表达风险”等信息。

对现实中同时具备 ToB 与 ToC 属性的用户，主路线只决定正式报告主体逻辑；但为了避免后续内容混讲，还需要额外判断：当前内容表达主场更适合站在哪一边、哪些内容可作为次路线补充、哪些内容应谨慎混讲。

若 route_type 判定为 ToB，但用户当前对外表达更容易从个人视角破圈，则可保留“主路线为 ToB、表达主场先借 ToC 壳子切入”的说明；反之亦然。但正式主体建议仍只能服从一条主路线，不得把两套逻辑混为一体。

十四、建议新增的补充判定字段

RT13｜primary_expression_arena｜单选｜TobPerspective / TocPerspective｜当前内容表达主场更适合站在哪一边。

RT14｜main_route_expression_rule｜文本｜主路线表达方式说明，例如“主体按 ToB 专业信任逻辑展开”。

RT15｜secondary_route_usage_note｜文本｜次路线只允许如何补充，不得如何混讲。

RT16｜mixed_expression_risk｜文本｜若混写 ToB / ToC，最可能出现的内容风险。

RT17｜launch_content_side｜单选｜主路线侧 / 次路线侧 / 混合过渡侧｜首发内容更适合从哪一侧切入。

十五、路由识别结果必须额外说明的内容

除 route_type、route_confidence、route_evidence、secondary_route_type 外，建议补充：当前内容表达主场更适合站在哪一边；什么是主路线表达；什么是次路线补充；哪些内容可以讲；哪些内容要谨慎混讲。

如果用户现实成交对象偏个人订阅，但洞察与案例来源明显带有 ToB 创始人服务背景，则不能只写“偏 ToC”或“偏 ToB”；还应把“主路线”“表达主场”“混讲风险”三件事分开写清。

如果路由明确但内容表达主场不明确，后续模块容易出现：前文像 ToB、后文像 ToC；平台建议和选题风格冲突；核心受众与首发内容不匹配。此类情况视为路线识别不完整。

十六、补充判定规则（用于后续IP内容表达）

主路线表达：指正式报告与主体建议必须服从的路线逻辑。ToB 主路线下，定位、栏目、平台、转化必须以企业客户、专业信任、案例教育为主；ToC 主路线下，必须以个人用户、痛点共鸣、传播性、人设记忆点为主。

次路线补充：指允许作为辅助素材出现的内容侧，但不得改变主体内容逻辑。例如 ToB 主路线下，可少量使用个人故事帮助破圈；ToC 主路线下，可少量引用专业案例抬高信任。

混讲风险：若用户同时具备两种属性，但没有明确“什么能混、什么不能混”，最常见的风险是账号定位摇摆、平台错配、选题风格冲突、转化路径断裂。

首发切入侧：除主路线外，还应判断首发内容更适合从主路线直接切入，还是先从次路线可传播内容切入，再逐步回归主路线。

十七、增强后的路由输出格式建议

{  "state_id": "S0.5",  "route_type": "TobFounder | TocPersonal",  "route_confidence": 0,  "route_evidence": ["..."],  "secondary_route_type": "TobFounder | TocPersonal | None",  "primary_expression_arena": "TobPerspective | TocPerspective",  "main_route_expression_rule": "主体报告如何服从主路线",  "secondary_route_usage_note": "次路线只允许如何补充",  "mixed_expression_risk": "若混讲最容易出现的风险",  "launch_content_side": "主路线侧 / 次路线侧 / 混合过渡侧",  "route_final_decision": "pass | need_more_info | reroute"}

十八、增强版自查补充

是否已经不仅判定了 ToB / ToC，还判定了“内容表达主场”。

是否已经说明主路线表达、次路线补充与混讲风险。

若首发内容更适合从次路线侧切入，是否已经说明这是“切入策略”，而不是主体路线变更。

十九、路线识别后与首批用户收窄的联动规则

路线识别通过后，不应只停留在 ToB / ToC 主路线判定，还需进一步输出“首批最核心成交对象、次级可覆盖对象、当前不优先服务对象”。

若主路线为 ToC，但用户的经验、案例、信任来源大量来自 ToB 服务背景，则应明确：ToB 经验主要作为厚度和信用背书，不得改变首批主受众。

若主路线为 ToB，但用户更容易从个人故事或个人视角切入传播，也必须说明这只是表达切入口，不等于改变主体成交对象。

二十、建议新增的首批受众与表达边界字段

RT18｜first_priority_audience｜文本｜首批最核心成交对象。

RT19｜secondary_coverable_audience｜文本或数组｜次级可覆盖对象。

RT20｜deprioritized_audience｜文本或数组｜当前不优先服务对象。

RT21｜route_expression_boundary｜文本｜主路线表达边界说明。

RT22｜credibility_backing_side｜文本｜次路线更适合承担哪类信用背书。

二十一、路线识别结果必须继续回答的关键问题

这个人的内容表达主场更适合站在哪一边。

主路线用于吸引谁、次路线用于证明什么。

哪些内容可以借次路线增加厚度，哪些内容一旦混讲就会导致受众摇摆。

首发内容更适合直接从主路线切入，还是从次路线更具传播性的壳子切入后再回归主路线。

二十二、路线识别结果向下游报告传递的最小要求

route_type 之外，至少还应向下游模块传递：primary_expression_arena、main_route_expression_rule、secondary_route_usage_note、mixed_expression_risk、launch_content_side、first_priority_audience。

若这些字段缺失，模块4 的受众收窄、模块9 的首发切入与模块10 的启动建议容易出现方向正确但表达主场不稳的问题。

二十三、路线识别后的技巧库自动挂载规则

从本版开始，route_type 判定不仅决定正式报告的主体写法，还必须直接决定后续调用哪套创作技巧库。主路线判定通过后，系统应自动挂载与主路线一致的创作技巧库，作为定位判断、内容方向、平台建议、转化建议、账号包装与执行规划的正式约束源。

route_type = TobFounder → 自动挂载 ToB 创作技巧库。

route_type = TocPersonal → 自动挂载 ToC 创作技巧库。

若存在 secondary_route_type，只能作为补充素材来源，不得改变主技巧库挂载方向。

二十四、建议新增的技巧库挂载字段

RT23｜skill_library_source｜文本｜当前挂载的技巧库来源，例如 ToB 创作技巧库 / ToC 创作技巧库。

RT24｜skill_library_status｜单选｜loaded / missing / fallback。

RT25｜skill_library_route_match｜单选｜pass / mismatch。

RT26｜skill_library_fallback_note｜文本｜技巧库缺失或不匹配时的降级说明。

二十五、路线与技巧库的一致性规则

主路线必须和技巧库路线一致。若 route_type = ToB，但 skill_library_source 挂载为 ToC 技巧库，则视为高风险 mismatch。

若 route_type 已通过，但没有 skill_library_source，则正式生成时必须进入 fallback 状态，并写清是按哪套默认路线规则降级生成。

若报告主体明显没有体现对应路线技巧库风格，即使 route_type 正确，也应视为路线识别后的生成约束未真正生效。

二十六、技巧库缺失时的 route 后降级说明

当 skill_library_status = missing 时，系统仍可继续生成，但必须在 route 后说明中写清：当前缺少对应技巧库支持，已按主路线默认规则降级生成。

fallback 状态下，允许保留主路线，但不得伪装成已经使用了某路线技巧库的细化逻辑。

fallback 状态应传递到下游模块，尤其影响模块4、模块5、模块6、模块8、模块9、模块10 的置信度说明。

二十七、路线识别结果向下游模块传递的最小技巧库字段

除 route_type、route_confidence、route_evidence、primary_expression_arena 外，至少还应向下游模块传递：skill_library_source、skill_library_status、skill_library_route_match、skill_library_fallback_note。

若这些字段缺失，下游模块虽然可能方向正确，但很容易只按通用逻辑写，无法体现 ToB / ToC 技巧库风格差异。

主路线识别不仅决定报告方向，也决定后续调用哪套创作技巧库；若只判路由、不挂技巧库，视为路由链条不完整。

阶段1分支、回流与输出对象补充

若 route_final_decision = need_more_info，则必须停留在本阶段补 route_evidence，不得进入 Demo 生成。

若主路线清晰但 primary_expression_arena、main_route_expression_rule 或 mixed_expression_risk 仍模糊，则先补“内容表达主场”判断，再进入下一阶段。

若 skill_library_status = missing 或 skill_library_route_match = mismatch，则进入 fallback 路径，并在 route_context 中显式写清降级说明。

本阶段完成后，至少输出并冻结：route_type、route_confidence、route_evidence、secondary_route_type、route_final_decision、primary_expression_arena、main_route_expression_rule、secondary_route_usage_note、mixed_expression_risk、skill_library_source、skill_library_status、skill_library_route_match、skill_library_fallback_note。

进入下一阶段前，必须确认：主路线已锁定、表达主场已明确、技巧库约束源已挂载或已记录 fallback。
